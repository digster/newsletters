1. Observability, monitoring, and logging
The leading observability vendor in this space, Datadog, was founded in 2010 with a focus on server and infrastructure monitoring: collecting metrics from servers, cloud instances, and with AWS as a focal point. Since then, Datadog has continued adding new products, and now supports dozens of features from APM (Application Performance Monitoring – distributed tracing, service maps), through RUM (Real User Monitoring – frontend performance, user sessions), to CI/CD visibility (pipeline monitoring, test analytics), and dozens more products. Datadog lists more than 50 on its website, and most are integrated to work together.
Other platforms are similar: they started with a narrow focus, added more capabilities, and became the full-blown platforms of today. A good example is Sentry: it started in 2008 as an open source monitoring tool for the Django platform in Python, then got incorporated in 2015 as a company, and today has products for APM (Application Performance Monitoring), error monitoring, session replay, and more.
With the caveat that comparing vendors in this crowded space is no simple task, below are listed the most-mentioned observability platforms by 3,000 respondents. (Thanks again to everyone who filled in the survey!):
Open source tools: several of these tools are open source (meaning they have permissive licenses):
Grafana [ https://substack.com/redirect/e4fe3fce-6707-4403-b641-7e6c74f4c011?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open source tool for displaying dashboards for observability. It’s often used with Prometheus, Elasticsearch, and other backend observability databases. Grafana Labs (creator of the project) offers a managed service and additional products like Loki and Tempo.
Prometheus [ https://substack.com/redirect/5e5716c0-1999-42f5-9e63-72cecd737be7?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source metrics monitoring system originally built at SoundCloud. It’s the de facto standard for Kubernetes monitoring, and often paired with Grafana for visualization.
OpenTelemetry [ https://substack.com/redirect/c382b5bf-69e8-453b-902e-ab56b3390ca7?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source observability framework providing standardized APIs, SDKs, and tools for instrumenting applications to collect traces, metrics, and logs.
Elasticsearch [ https://substack.com/redirect/09354f5d-aee6-488a-90e3-6030391c8536?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ] is an open-source distributed search and analytics engine, designed for storing and querying large volumes of data in near-real time. Elasticsearch moved from being open source in 2021 [ https://substack.com/redirect/d36ba78c-d2af-487d-8e6f-d68719f33325?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ] to a more restrictive license, before returning to open source in 2024 [ https://substack.com/redirect/6f6247e1-1d4e-4a6c-ad0b-931313ad5a68?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ].
Loki [ https://substack.com/redirect/8e90d1b3-45c3-44cf-b944-db8ec1f53697?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source log aggregation system created by Grafana Labs, inspired by Prometheus and designed for logs, instead of metrics. Integrates nicely with Grafana for visualization and with Prometheus for unified observability.
Tempo [ https://substack.com/redirect/f3f1a203-2bd2-445b-8d2b-49806f92ea73?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source distributed tracing backend that’s designed for massive scale and cost-effectiveness by Grafana Labs. It integrates nicely with Grafana and Prometheus.
Jaeger [ https://substack.com/redirect/209d2ee2-161c-42e2-bc03-3c3531dd548f?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source distributed tracing system originally developed by Uber for monitoring microservices-based architectures. Commonly used as a backend for OpenTelemetry tracing data.
Zabbix [ https://substack.com/redirect/ef31adae-fe2a-4a5a-b3c7-26873caf3651?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source enterprise monitoring solution for networks, servers, and infrastructure components that’s popular in more traditional environments.
SigNoz [ https://substack.com/redirect/2b5e6ccf-c9b2-4dfe-8cd9-9fd8ec689c9a?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ]: an open-source, full-stack observability platform that provides APM (Application Performance Monitoring), distributed tracing, metrics, and logs in a single application. Built as an open-source alternative to commercial platforms like Datadog or New Relic, using OpenTelemetry for instrumentation.
Grafana coupled with Prometheus is a popular combination for companies to build an in-house observability stack. This is what Shopify did [ https://substack.com/redirect/617f257f-6db5-48d7-ad8e-09ea5f59b9cf?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ] when it moved off Datadog – and it was also the stack which Coinbase built after receiving an eye-watering $65M bill from Datadog in 2023. A new generation of incident management tools appears to be challenging the old guard. PagerDuty was founded in 2009 and OpsGenie in 2012, and these two tools are still the most mentioned in our survey. However, a group of startups is emerging as challengers:
incident.io: founded 2021
Rootly: founded 2020
FireHydrant: founded 2018
What seems to have shaken things up and helped the “new generation” of incident management startups spread, was PagerDuty and OpsGenie being slow to add support for handling incidents inside Slack. incident.io, Rootly, and FireHydrant gained initial momentum by offering this.
Today, pretty much all the above oncall tooling vendors describe themselves on their landing pages as “AI platforms” or “AI-first operations platforms”, as they’re experimenting with new features built on top of LLMs like incident summaries, or taking a first stab at root cause analysis using LLMs.

What stands out:
Feature flagging. According to our survey, the type of system that’s most often custom-built in house is for feature flagging. A/B testing and experimentation. Another common mention. Integrating A/B testing into internal systems – such as tools for the data team to analyze results, or to integrate with the feature flag system – is a common reason to build in-house.
CI/CD systems. Using custom continuous integration / continuous deployment systems is common enough, although it’s more common to integrate software like Jenkins [ https://substack.com/redirect/ec5f1306-96d4-4574-8166-a466c2cc5013?j=eyJ1IjoiOG81NG4ifQ.6oeetudtJEh-zO1onUJvtadWEdbodFIw0h4xbugrh1o ] and run it on existing infrastructure, instead of building “truly” custom software.

Takeaways
I hope you’ve found this article and the two others in this mini-series about our survey results useful. What stands out for me in this final set of responses is:
Observability and logging are tough problems. Leading vendors offer a large number of products that integrate with each other, but open source products for personally building and operating an observability stack are equally popular.
Of course, there is no single ideal solution: every team needs to figure out which tools work well enough for them.
I sense less innovation in frontend and mobile these days. React is as popular as ever, and Angular and Vue are the stable next choices for a minority of developers. Across Android and iOS, the most popular frameworks mentioned were released in 2019 (SwiftUI for iOS, Jetpack Compose for Android).
Developer portals’ momentum seems to have stalled. Backstage was mentioned by 1 in 60 respondents, suggesting most companies don’t have service discoverability issues, or they solve it with internal, custom tools.
As developers, we can (and do!) build custom tools to solve problems. I really enjoyed learning from readers about their custom, internal tools built for problems like feature flagging, CI/CD, and others. Some tech leaders and executives might argue that building in-house is not a great tradeoff, and ask why not just buy solutions from vendors.
But at the end of the day, we’re engineers.
Sometimes, when a good-enough solution isn’t available, we are more than capable of building our own! If we always defaulted to choosing a vendor, we’d miss out on the pleasure of sharpening our skill at building from scratch.
